Руководство для AI-ассистентов, работающих с репозиторием SamorodinkaTech.Fiducia.

## Обзор проекта

**Цифровой Совет Директоров** — платформа электронного управления заседаниями совета директоров для АО и ООО. Полное соответствие 208-ФЗ, 14-ФЗ, 63-ФЗ, 152-ФЗ.

**Технологический стек**:
- .NET 8 / C# (ASP.NET Core)
- PostgreSQL 16
- Redis 7
- RabbitMQ 3.12
- Docker / Docker Compose

**Ключевые особенности**:
- Два портала: Board Portal (для директоров) и Admin Console (для администраторов)
- Офлайн-функционал для голосования без интернета
- ГОСТ-генерация документов (ГОСТ Р 7.0.97-2025)
- Некорректируемый аудит-лог для ИБ
- Динамические комитеты с двумя типами логики

```bash

dotnet build

# Запуск API
dotnet run --project src/Api

# Запуск с hot-reload
dotnet watch --project src/Api
```

```bash
# Все тесты
dotnet test

# Unit тесты
dotnet test tests/Unit

# С покрытием
dotnet test /p:CollectCoverage=true /p:CoverletOutput=./coverage/
```

## Правила запуска приложения для пользователя (КРИТИЧНО)

- Если пользователь просит **запустить сайт/приложение** для себя (чтобы посмотреть в браузере), запуск должен выполняться в **фоновом режиме** (`waitForCompletion: false` для `run_terminal_command`) и **не должен завершаться (killed) ассистентом** после диагностики или проверки.
- Запрещено использовать сайт, поднятый "для своих нужд" (диагностика, тестирование, verification), как единственный запущенный экземпляр, а затем гасить его — если пользователь ожидает доступ к запущенному сайту, нужно поднять/оставить **отдельный, стабильно работающий процесс**.
- После запуска приложения по просьбе пользователя:
  1. Убедиться, что процесс поднялся и порт слушается (например, через `curl -sI http://localhost:<port>`).
  2. Открыть сайт в **Google Chrome** (например, `open -a "Google Chrome" http://localhost:<port>`).
  3. Сообщить пользователю адрес и порт, а также команду для остановки процесса, когда это потребуется.
- Порты фиксированы и не имеют синонимов/алиасов — каждый хост слушает ровно один порт:
  AdminConsole — **5001**, BoardPortal — **5002**. Не вводить альтернативных портов для одного хоста
  (HTTPS-редирект отключён в Dev-режиме, см. правило ниже).
  Порт 5000 зарезервирован macOS (AirPlay Receiver) и недоступен на машинах разработчиков.
- **Важно**: порты не искать в `launchSettings.json` — они могут содержать устаревшие или
  конфликтующие значения. Запуск обоих порталов — `./start.sh`.
- Board Portal запускается с переходом на страницу входа: `http://localhost:5002/login`.
- Admin Console запускается с переходом на страницу входа: `http://localhost:5001/login`.
- **По умолчанию**: если пользователь просит запустить сайт без уточнения — запускается Board Portal (порт 5002).
- Диагностические/временные запуски (для отладки, скриптов, Playwright и т.п.) должны выполняться в отдельных процессах/портах и не подменять или останавливать сайт, запущенный для пользователя.

## Дисциплина работы ассистента (КРИТИЧНО)

### Причины сбоев (инцидент, зафиксировано в истории проекта)

Зафиксированы повторяющиеся сбои исполнения с общей корневой причиной —
результат вызова инструмента принимался на веру без независимой проверки
фактического состояния файла, а также основная задача пользователя
подменялась сопутствующими мета-правками. Конкретные проявления:

1. Инструмент редактирования возвращал "успех", но изменения не сохранялись
   в файле — это не проверялось повторным чтением файла в том же ответе.
2. Внимание переключалось на правки документации/правил вместо выполнения
   основного запроса пользователя, из-за чего основная задача откладывалась
   на неопределённый срок.
3. Инструменты вызывались с пустыми, некорректными или плейсхолдерными
   аргументами (`...`, `TODO`, `REPLACED_PLACEHOLDER`) вместо реального
   содержимого.
4. Исполнение останавливалось без наличия реального вопроса к пользователю,
   ответ на который менял бы дальнейшие действия — либо, наоборот, реальный
   вопрос пользователя (например, "где мы отобразим бизнес-уровень?")
   игнорировался, и ассистент сразу переходил к правке файлов без ответа.

### Правило: сначала ответ на вопрос, потом действие

Если пользователь задал явный вопрос (например, о месте/способе отображения
чего-либо в архитектуре) — сначала даётся содержательный ответ на этот
вопрос в тексте, и только после этого (или вместе с этим) выполняются
правки файлов. Молча переходить к правкам файлов, игнорируя заданный
вопрос, запрещено.

### Правило: приоритет основной задачи

Основной запрос пользователя выполняется первым и до конца. Любые
сопутствующие мета-правки (документация, правила, рефакторинг "по пути")
выполняются только после завершения основной задачи и не должны её
подменять или откладывать.

### Правило: уточнение затрагиваемых модулей перед реализацией (scope creep)

Зафиксирован инцидент: пользователь просил сделать настраиваемые пути
логирования, но не просил добавлять это в пользовательский интерфейс —
тем не менее реализация вышла за рамки запроса.

Чтобы это не повторялось:

- Перед реализацией любой задачи, затрагивающей более одного модуля
  (например, `Api`, `Application`, `Domain`, `Infrastructure`, Board Portal,
  Admin Console), необходимо явно уточнить (себе — через диагностику кода,
  либо у пользователя, если неоднозначно) **полный список всех
  затрагиваемых модулей системы** и зафиксировать его перед началом правок.
- Реализация ограничивается ровно тем скоупом, который запросил
  пользователь. Расширение скоупа (например, добавление UI-настроек,
  когда просили только backend-конфигурацию) запрещено без явного запроса
  пользователя.
- Если по ходу диагностики становится ясно, что задача логически требует
  изменений в дополнительном модуле, это выносится в текстовый ответ
  пользователю как отдельный пункт (предложение/вопрос), а не реализуется
  молча "заодно".

### Правило: диагностика сложной логики перед модификацией (КРИТИЧНО)

Зафиксирован инцидент: при добавлении чекбоксов типов директоров в форму ОСА
был **без диагностики переписан** метод `EnsureBoardMemberCount`, затронуты
смежные методы `OnBoardCompositionChanged`, `OnTempChairChanged` и
`ApplyTempChairRole`. Это привело к каскадным поломкам: временный
председательствующий перестал назначаться, таблица «Готовимся к первому
заседанию» откатилась к одной строке без типов, чекбоксы директоров
сбрасывались. Корневая причина — не было прочитано и понято **всё дерево
вызовов** между затронутыми методами (`@bind:after` → callback → метод → 
смежные методы → дополнительные обработчики) до начала правок.

Правило:
- Перед модификацией метода, который имеет **сложную логику** (3+ уровня
  вложенности, перекрёстные вызовы, несколько точек входа через `@bind:after`,
  `async void`-обработчики), необходимо:
  1. Прочитать **все** методы, связанные с изменяемым (вверх и вниз по стеку
     вызовов: кто вызывает → что вызывает → какие методы изменяет).
  2. Построить (мысленно или в тексте) граф зависимостей: «метод A вызывает
     B и C, B изменяет поле X, C зависит от поля X».
  3. Если граф неполон или есть риск побочного эффекта в несмежной области —
     **спросить пользователя** перед правками, описав риски.
- Запрещено переписывать метод «с нуля», если можно обойтись **минимальной
  хирургической правкой** (добавить условие/вызов, а не заменять весь метод).
- Если метод уже был отлажен и работает в production — любое его изменение
  требует **явного подтверждения пользователем** предлагаемого подхода.

### Правило: ищи прямое, а не обратное

Зафиксирован инцидент: при поиске строки для временного председательствующего
использовался фильтр «первая не-секретарская строка» (`RoleId != secRoleId`)
вместо прямого поиска строки с ролью TEMP_CHAIR (`RoleId == tcRoleId`), а также
`boardMemberRows[0]` вместо `FirstOrDefault(role == TEMP_CHAIR)`. Это привело
к тому, что секретарь перехватывал позицию, роль не назначалась, и потребовалось
7+ итераций правок для исправления.

Правило: если нужно найти/добавить/удалить X — ищи X напрямую, а не «всё кроме Y».
- Не `FirstOrDefault(r => r.RoleId != секретарь)`, а `FirstOrDefault(r => r.RoleId == TEMP_CHAIR)`
- Не `boardMemberRows[0]` (предполагая, что первый — не секретарь),
  а `FirstOrDefault(r => r.RoleId == TEMP_CHAIR)`
- Прямой поиск исключает побочные эффекты от изменения порядка элементов в списке.

### Правило: раздельные страницы вместо объединённых (КРИТИЧНО)

По возможности каждая сущность/справочник оформляется **отдельной страницей**,
а не объединяется с другими на одной странице с вкладками.

- Объединение нескольких справочников в одну страницу с вкладками — только
  по явному запросу пользователя или после явного предложения с его согласия.
- Не проявлять инициативу по объединению страниц, если пользователь
  не просил об этом.
- При добавлении нового справочника — отдельная страница и отдельный пункт
  меню (если пользователь не указал иное).

### Правило: единственный вызов ApplyConfigurationsFromAssembly (КРИТИЧНО)

Зафиксирован инцидент: в `OnModelCreating` метод `ApplyConfigurationsFromAssembly`
был вызван дважды (строки 54 и 313), из-за чего seed-данные из
`IEntityTypeConfiguration<T>.HasData()` регистрировались повторно с теми же
ключами. Ошибка: `"The seed entity for entity type 'X' cannot be added because
another seed entity with the same key value for {'Id'} has already been added"`.

Правило: `ApplyConfigurationsFromAssembly` (и любые другие методы загрузки
конфигураций из сборки) вызывается **ровно один раз** в `OnModelCreating`.
- Повторный вызов вызывает дублирование seed-данных и другие трудно-диагностируемые
  ошибки на этапе инициализации модели EF Core.
- Если в методе уже есть вызов `ApplyConfigurationsFromAssembly` — не добавлять
  второй даже «для надёжности»; удалить дубликат при обнаружении.
- Исключение: **разные** сборки с **непересекающимся** набором типов —
  но и в этом случае рекомендуется единый вызов с типом из сборки,
  содержащей все конфигурации.

### Правило: дисциплина вызова инструментов

- Каждый вызов инструмента — это настоящий вызов с реальными аргументами,
  а не текстовый пример или плейсхолдер.
- Плейсхолдеры (`...`, `TODO`, `REPLACED_PLACEHOLDER` и т.п.) внутри
  аргументов реального вызова запрещены.
- После каждого вызова, изменяющего файл, результат немедленно проверяется
  повторным чтением файла в том же ответе, прежде чем сообщать об успехе.
- Если проверка показала, что изменение не применилось — попытка немедленно
  повторяется с точным содержимым, без переспрашивания пользователя.

### Правило: экранирование кавычек в shell-командах (zsh, `run_terminal_command`)

Зафиксирован повторяющийся сбой: команда для терминала оборачивалась во
внешние двойные кавычки `"..."` вместе с внутренними экранированными
кавычками (например, `"git commit -m \"текст\n\nтекст\""`), из-за чего zsh
падал с ошибкой `no such file or directory` или `command not found`,
пытаясь интерпретировать всю строку как путь к файлу. Ошибка повторялась
дважды подряд в одной сессии, включая попытку исправить её той же неверной
конструкцией (внешние кавычки вокруг `grep -n '...'`).

Правила во избежание этого:
- Аргумент `command` инструмента `run_terminal_command` передаётся **без**
  оборачивания всей команды в дополнительные внешние кавычки — только те
  кавычки, которые нужны самой shell-команде (например, `git commit -m "текст"`
  или `grep -n 'паттерн' file`), без обёртки вида `"grep -n '...' file"`.
- Для многострочных сообщений коммита использовать **несколько флагов**
  `-m "строка1" -m "строка2"` вместо `\n` внутри одной строки — это надёжнее
  и не требует экранирования переносов строк в zsh.
- Если команда с кавычками возвращает `no such file or directory` или
  `command not found` — это почти всегда сигнал об ошибке экранирования,
  а не о том, что файла/бинаря не существует; в первую очередь нужно
  упростить кавычки, а не менять путь или диагностировать окружение.
- После получения ошибки shell-синтаксиса запрещено вслепую повторять
  структурно ту же конструкцию с кавычками — нужно изменить сам подход
  к экранированию перед повторной попыткой.

### Правило: миграции только по запросу пользователя

- Не генерировать миграционные скрипты (EF/SQL) и не предлагать их генерацию,
  пока пользователь явно не попросит об этом в задаче.

### Правило: автономность исполнения

Остановка исполнения допустима только при наличии реального вопроса к
ассистенту от пользователя или необходимости уточнить у пользователя
неоднозначную трактовку задачи, риск непоправимого действия, либо
недостающие данные, которые невозможно получить самостоятельно.
Диагностика (чтение файлов, `git status`, `grep`) — часть выполнения
задачи, а не повод для промежуточной остановки с отчётом.

### Правило: применение изменений БД после правок схемы

- Если задача включает добавление/изменение полей в EF-модели, изменения
  применяются к реальной БД немедленно (SQL-командой или миграцией) — без
  отдельного запроса разрешения у пользователя.
- Некорректно: закомитить код с новым полем, но не создать колонку в БД,
  после чего приложение падает с ошибкой при запуске.
- Если EF-миграция не может отработать инкрементально (нет истории), колонка
  добавляется напрямую через SQL и в БД, и в коде.
- Исключение: генерация нового миграционного скрипта (`dotnet ef migrations add`)
  выполняется только по запросу пользователя.

### Правило: восстановление ErrorBoundary при навигации (Blazor Server)

Зафиксирован инцидент: ошибка на странице А «переживала» навигацию и
показывалась на странице Б, поскольку `ErrorBoundary.Recover()` вызывался
в `OnParametersSet`, который срабатывает не только при навигации, но и при
любом изменении параметров.

Правило: в `MainLayout` (Admin Console и Board Portal) восстановление
`ErrorBoundary` выполняется через подписку на
`NavigationManager.LocationChanged`, а не через `OnParametersSet`.

Шаблон реализации:
```csharp
@implements IDisposable

protected override void OnInitialized()
{
    Navigation.LocationChanged += OnLocationChanged;
}

private void OnLocationChanged(object? sender, LocationChangedEventArgs e)
{
    _errorBoundary?.Recover();
    StateHasChanged();
}

public void Dispose()
{
    Navigation.LocationChanged -= OnLocationChanged;
}
```

- `LocationChanged` гарантирует вызов только при реальном переходе между
  страницами, а не при каждом цикле рендеринга.
- Обязательно `StateHasChanged()` после `Recover()`, чтобы форсировать
  перерисовку и не оставлять страницу в состоянии ошибки.
- Реализовать `IDisposable` для корректной отписки. Без этого возможна
  утечка обработчиков и повторные срабатывания.

### Правило: отключение HTTPS-редиректа при локальной разработке (КРИТИЧНО)

Зафиксирован инцидент: пользователь явно просил перевести Board Portal на HTTP,
ассистент убрал `launchBrowser` из `https`-профиля в `launchSettings.json`,
но HTTP-запросы всё равно редиректились на HTTPS. Корневая причина —
`app.UseHttpsRedirection()` в `Program.cs` вне блока `!IsDevelopment()`.

Правило: при переводе приложения на HTTP для локальной разработки
недостаточно изменить `launchSettings.json` — необходимо проверить и
исправить **оба** источника:

1. **launchSettings.json** — HTTPS-профиль (`https`) не должен автоматически
   открывать браузер (`launchBrowser: true`), если целевой режим — HTTP.
2. **Program.cs** — `app.UseHttpsRedirection()` должен находиться внутри
   блока `if (!app.Environment.IsDevelopment())`, вместе с `UseHsts()` и
   `UseExceptionHandler()`.

Без второго пункта middleware продолжит редиректить HTTP → HTTPS независимо
от конфигурации запуска.

## Архитектурные принципы

### Проектная карта

- При изменении архитектуры, функционала или конфигурации обновлять
  проектную карту: `README.md`, `AGENTS.md`, `docs/`.
- Это правило гарантирует, что документация не отстаёт от кода.

### ЮЛ vs Юридические лица (КРИТИЧНО)

- **«ЮЛ»** (пункт меню, страница `/admin/legal-entities`) — режим
  редактирования ОДНОГО выбранного юридического лица с его настройками
  (руководитель, Совет директоров, интервал ГОСА). Страница всегда
  работает с одной записью и **обновляет** её.
- **«Юридические лица» / `legal_entities`** — справочник (таблица БД),
  содержащий множество записей юрлиц (ПАО, НАО, ООО и др.).
- Запрещено: формулировки «создать ЮЛ», «сохранить ЮЛ как новую запись».
  Корректно: «выбрать юрлицо из справочника», «сохранить настройки
  выбранного юрлица».
- Режим «ЮЛ» **никогда не создаёт** новые записи в `legal_entities`.

### Целевой фреймворк и общий код

- Target framework: `net9.0` во всех проектах решения.
- Два портала (Board Portal и Admin Console) делят идентичный код;
  изменения, затрагивающие общие компоненты, применяются к обоим порталам.

### Чистая архитектура

Зависимости направлены внутрь:

```
Api → Application → Domain ← Infrastructure
```

**Domain** не зависит от внешних библиотек.

### CQRS vs CRUD (важно про терминологию)

- **Не называть CQRS** стандартный CRUD-шаблон (репозиторий/сервис с методами Create/Read/Update/Delete поверх одной модели и одной БД). Это просто разделение методов по действию, а не паттерн CQRS.
- Настоящий **CQRS** требует **логического разделения верхнеуровневых модулей** для команд (Commands) и запросов (Queries): отдельные модели, отдельные пути обработки (разные namespace/классы/handlers), а в идеале — **раздельные хранилища/инстансы БД** для чтения и записи (Read replica / отдельная read-модель), с явной синхронизацией между ними (например, через Domain Events).
- Если в модуле просто есть класс с методами `CreateAsync`, `GetAsync`, `UpdateAsync`, `DeleteAsync` над одной и той же таблицей/DbContext — это **CRUD**, и называть его CQRS в коде, документации, PR и коммитах **запрещено**.
- **Commands**: Запись данных (Create, Update, Delete) — в терминологии CQRS.
- **Queries**: Чтение данных (Get, Search) — в терминологии CQRS.

### Конфигурация логирования (КРИТИЧНО)

- Все настройки логирования (уровни логирования, пути к файлам логов,
  форматы, sink'и, ротация) хранятся **исключительно в конфигурационном
  файле** (`appsettings.json` / `appsettings.{Environment}.json` или
  переменные окружения), а не в базе данных и не в пользовательском
  интерфейсе (Admin Console / Board Portal).
- Изменение путей логирования и других параметров логирования через UI
  запрещено, если это не было явно запрошено пользователем отдельно.
- Любая реализация настраиваемых путей/параметров логирования должна
  использовать стандартный механизм конфигурации ASP.NET Core
  (`IOptions<TLoggingOptions>` / `IConfiguration`), без создания
  дополнительных административных экранов для их редактирования.

### Конфигурация внешних интеграций (ADR-022)

- Все настройки внешних интеграций (URL API, ключи доступа, таймауты,
  флаги включения/отключения) хранятся **исключительно в конфигурационном
  файле** (`appsettings.json` / переменные окружения).
- Хардкодинг URL, ключей или endpoints в коде запрещён.
- Хранение настроек интеграций в базе данных и их редактирование через UI
  запрещено.
- Рекомендуемый способ чтения: `IOptions<T>` с привязкой секции конфига
  (для интеграций с 3+ параметрами) или `IConfiguration["Section:Key"]`
  (для простых случаев).
- Правило действует для всех интеграций: TrueConf, МТС Линк, LDAP,
  FileStorage, СПАРК, SecurityAudit, Session, Serilog.

### Инициализация внешних интеграций (ADR-023)

- Конструкторы клиентов внешних интеграций **не выполняют сетевых соединений**.
  Подключение к внешнему сервису — при первом вызове метода (lazy), а не в конструкторе.
- Ошибки конфигурации (невалидные значения в `appsettings.json`) **не должны крашить
  приложение**. Вместо `throw new InvalidOperationException` используется
  `Log.Warning` + fallback на значение по умолчанию или отключение интеграции.
- Опциональные интеграции регистрируются только при явном флаге `Enabled` /
  непустом ключе доступа. Отсутствие конфигурации для опциональной интеграции —
  не ошибка.
- Для критичных параметров безопасности (JWT-секрет) в production сохраняется
  `throw`; в dev-окружении допускается dev-ключ с предупреждением в логах.
- Рекомендуемый паттерн для будущих интеграций с тяжёлой инициализацией:
  `Lazy<T>` в DI с try-catch и fallback на no-op реализацию.

### Формат имён лог-файлов

- Файлы приложения: `app-{yyyyMMddHH}.log` (Hourly, ротация каждый час).
- Файлы аудита: `audit-{yyyyMMddHH}.log` (Hourly).
- Тестовые логи: `{type}-test-{yyyyMMdd}.log` (Daily).
- Все имена содержат часы для оперативного мониторинга и поиска инцидентов.

### Серверная валидация (BDR‑008) — логирование

- Вызов метода серверной валидации логируется как **Debug**:

  `LogWarning("Валидация не пройдена для {EntityType} {Id}: {Errors}")`.
- Правило действует для всех форм сохранения в Admin Console (BDR‑008).

## Правила кодирования

### Терминология (КРИТИЧНО)

- Запрещено использовать слово «функционал» для обозначения функций/возможностей системы.
  Это калька с английского *functional*. Корректный русский термин — **«функциональность»**.
  Пример: «добавить функциональность создания комитетов», а не «добавить функционал комитетов».
- Правило действует во всех артефактах проекта: код, комментарии, документация, сообщения коммитов, PR, интерфейс.

### Стиль C#

- Namespace: `Fiducia.{Layer}.{Module}`
- Classes: PascalCase (`MeetingService`)
- Interfaces: I + PascalCase (`IMeetingService`)
- Methods: PascalCase (`CreateAsync`)
- Parameters: camelCase (`cancellationToken`)
- Private fields: _camelCase (`_repository`)

### XML Documentation

Все публичные члены должны иметь XML-документацию:

```csharp
/// <summary>
/// Создаёт уведомление о созыве заседания совета директоров.
/// </summary>
/// <param name="request">Данные уведомления.</param>
/// <param name="cancellationToken">Токен отмены.</param>
/// <returns>Идентификатор созданного уведомления.</returns>
public async Task<int> CreateAsync(
    CreateMeetingRequest request,
    CancellationToken cancellationToken = default)
```

### Документирование полей EF-моделей

Все поля доменных сущностей (`src/Domain/Entities/*.cs`) должны иметь XML-комментарий с русским описанием и именем колонки в БД в скобках:

```csharp
/// <summary>Наименование юридического лица (name).</summary>
public string Name { get; set; } = default!;

/// <summary>Тип участника СД (board_member_type_id).</summary>
public Guid? BoardMemberTypeId { get; set; }
```

- Имя колонки в комментарии должно совпадать с `HasColumnName` в EF-конфигурации
- При переименовании поля или колонки комментарий обновляется немедленно
- Системные поля (`Id`, `CreatedAt`, `CreatedBy` и др.) именуются одинаково во всех сущностях без префиксов — они часть внутреннего устройства системы

### Dependency Injection

Внедрение зависимостей через конструктор:

```csharp
public class MeetingService : IMeetingService

    private readonly ILogger<MeetingService> _logger;
    private readonly ITimeProvider _timeProvider;

    public MeetingService(
        IMeetingRepository repository,

        _logger = logger ?? throw new ArgumentNullException(nameof(logger));
        _timeProvider = timeProvider ?? throw new ArgumentNullException(nameof(timeProvider));
    }
}
```

### Обработка ошибок (КРИТИЧНО)

- Пустые блоки `catch { }` или `catch { /* ignore */ }` **запрещены** во всём
  коде проекта — C#, Razor, скриптах и пр. Подавление ошибок без логирования
  скрывает истинную причину сбоев и приводит к недиагностируемым отказам
  (включая падение Blazor-цикла из-за необработанных исключений в `async void`).
- Каждый `catch` должен как минимум логировать исключение:
  ```csharp
  catch (Exception ex)
  {
      _logger.LogWarning(ex, "Описание контекста ошибки");
  }
  ```
- Если ошибка намеренно игнорируется (редкий случай), это должно быть явно
  обосновано комментарием с указанием причины: `// Допустимо: сетевой сбой
  при проверке недоступного внешнего сервиса, fallback есть выше`.
- Особое внимание — методам `async void`: необработанное исключение в них
  крашит процесс или Blazor-цикл. Каждый `async void` обязан иметь внешний
  `try/catch` с логированием на всём теле метода.
- Асинхронные методы, изменяющие состояние компонента (установка полей,
  переход между режимами — создание → редактирование), **обязаны вызываться
  через `await`** в обработчиках событий. Fire-and-forget (`Method()` вместо
  `await Method()`) приводит к тому, что изменения состояния происходят после
  завершения цикла рендеринга и UI не обновляется. После завершения таких
  методов рекомендуется вызывать `StateHasChanged()`.
- **Синхронный callback** (`void`-метод, вызываемый из `@bind:after` или `@onchange`):
  изменяет состояние компонента в том же цикле рендеринга. Вызов `StateHasChanged()`
  в конце метода ДОСТАТОЧЕН — `await InvokeAsync(StateHasChanged)` НЕ НУЖЕН и
  не компилируется в `void`-методе. Попытка использовать `await` в `void`-методе
  (включая `_ = InvokeAsync(...)`) — ошибка.
- **Асинхронный callback** (`async Task` / `async void`): если метод выполняет
  `await` (БД, HTTP), используется `await InvokeAsync(StateHasChanged)` для
  переключения на поток Blazor-рендерера.
- Запрещено смешивать: нельзя писать `await` в `void`-методе, нельзя в
  синхронном методе использовать `InvokeAsync` без причины.
- При логировании/показе ошибок БД — всегда извлекать最深нее сообщение из
  цепочки `InnerException`. EF Core оборачивает реальную ошибку в
  «An error occurred while saving the entity changes» — показывать нужно
  внутреннее исключение. Использовать `UnwrapException(ex)`, а не `ex.Message`.

### Blazor: `@bind` к элементу списка по индексу (КРИТИЧНО)

- Запрещено использовать `@bind="list[i].Property"` в `@for`-цикле.
  Индекс `i` захватывается на момент рендеринга; при модификации списка
  индекс устаревает — биндер падает с `ArgumentOutOfRangeException`.
- Вместо этого: захватить ссылку `var row = list[i]` и биндить к `row.Property`.
  Ссылка на объект остаётся валидной даже при перестроении списка.
- То же касается `@onchange="e => Handler(i, e)"` — передавать объект `row`,
  а не индекс.

### Blazor: `invisible` вместо `@if` для условно-видимых sibling-элементов в строке (КРИТИЧНО)

- `@if (condition) { <элемент> }` добавляет/убирает элемент из DOM, вызывая
  вертикальный скачок строки (layout shift) при переключении чекбокса.
- Паттерн: элемент рендерится **всегда**, но переключается через Bootstrap-класс
  `invisible` (`visibility: hidden`, сохраняет место в layout):
  `<select class="form-select form-select-sm @(condition ? "" : "invisible")" ...>`
- Гарантирует неизменную геометрию строки независимо от состояния чекбокса.
- Пример из проекта: поля количества директоров в левой колонке «Штатный состав СД»,
  дропдаун временного председательствующего, чекбокс «секретарь подписывает протоколы».

### Blazor: radio-кнопки через `@onclick` на родителе, а не `@onchange` на `<input>` (КРИТИЧНО)

- `checked` + `@onchange` на `<input type="radio">` **не работает** в Blazor:
  фреймворк управляет атрибутом `checked` и откатывает изменение DOM до того,
  как срабатывает C#-обработчик `@onchange`.
- Паттерн: `@onclick` на родительском `<div class="form-check">`, а `checked`
  используется только для отображения текущего состояния:
  ```html
  <div class="form-check" @onclick="() => _field = SomeEnum.ValueA">
      <input class="form-check-input" type="radio" name="group"
             checked="@(_field == SomeEnum.ValueA)">
      <label class="form-check-label">Вариант А</label>
  </div>
  ```
- Метка (label) и radio-кружок должны быть нажимаемыми — клик по всей области
  `<div class="form-check">` меняет значение.
- Пример из проекта: выбор вида документооборота во вкладке «Документооборот»
  режима ЮЛ.

### EF Core: `HasDefaultValue` для enum — через значение enum, не через `int`

- `HasDefaultValue(0)` для enum-свойства падает на этапе design-time с ошибкой
  «Cannot set default value '0' of type 'int' on property of type 'Enum'».
- Правильно: `HasDefaultValue(DocumentFlowType.Paper)` — передавать значение
  самого enum, а не числовой литерал.

### Схема БД: проверять соответствие EF-маппинга реальной структуре таблиц

- При изменении EF-маппинга (или при первом обращении к таблице) сверять:
  имя таблицы (`ToTable`), первичный ключ (`HasKey`), имена колонок (`HasColumnName`),
  имена индексов — с реальной структурой в БД (`\d table_name`).
- Несовпадение приводит к ошибкам `42P01: relation does not exist` или неверной
  работе связей.
- Пример: `OkopfConfiguration` была на таблицу `okopf` с PK по `Code`,
  а в БД таблица `ref_okopf` с PK `Id`.

### Валидация: зеркалирование клиент-сервер (КРИТИЧНО)

- Клиентская валидация (`IsStepValid` / `AreXxxFieldsValid`) обязана
  проверять **все** условия, которые проверяет серверная (`SaveEdit`).
  Частичная проверка приводит к тому, что вкладка остаётся зелёной,
  а `stepReachable` не ломается — ошибка обнаруживается только при
  финализации.
- Пример: `IsProtocolStepValid()` проверяет не только «подписан + дата есть»,
  но и «дата в допустимом диапазоне».

### Валидация: разделение form-валидатора и DB-валидатора (КРИТИЧНО)

- **Form-валидатор** (поля объекта) — проверяет значения полей текущей формы:
  диапазоны, обязательность, совместимость. Не обращается к БД.
  Пример: `OsaMeetingValidator` — проверяет `ShareholdersCount`, `GosaYear`,
  конфликт «заочное + интервал ГОСА».
- **DB-валидатор** (уникальность по БД) — проверяет бизнес-правила, требующие
  выборки из БД: уникальность записей, существование связанных сущностей.
  Пример: проверка «не более 1 ГОСА в году» через `AnyAsync(...)`.
- Эти два типа валидации реализуются в **разных классах**.
- В unit-тестах form-валидатор и DB-валидатор тестируются в **разных файлах**:
  `*ValidatorTests.cs` (чистые тесты полей) и `*BusinessRulesTests.cs`
  (тесты с `FakeDbContext`).
- Для mock-объекта БД (`FakeDbContext`) **рекомендуется отдельный тестовый проект**
  (например, `Tests.Unit.Db`), чтобы явно отделить инфраструктурные тесты от
  тестов доменной логики.

### Сохранение: мягкое vs жёсткое

- «Сохранить» (промежуточное) — сохраняет без валидации, не блокирует.
- «Завершить» (финализация) — полная валидация, блокирует при ошибках.
- `QuickSave` вызывается при любом переходе между вкладками — данные
  не должны теряться при навигации по wizard.

### Данные формы: не дублировать и не изобретать поля

- Прежде чем добавлять новое поле в сущность/БД — проверить, не решает ли
  задачу уже существующее поле.
- Пример: для «даты проведения заседания» использовать существующий
  `GosaWindowStart`, а не добавлять `MeetingDate`.

### Список и счётчик: пересчитывать при каждом входе на вкладку

- При переходе на вкладку с таблицей, размер которой зависит от другого поля
  (например, `editBoardMemberNumber`), необходимо **пересчитывать количество
  строк**, а не только загружать из БД. БД может содержать устаревшее количество.
- Метод `EnsureBoardMemberCount()` добавляет/убирает строки до совпадения
  с заявленным количеством.

- **FluentAssertion**: Утверждения

### DB-валидаторы: тестирование через Moq (КРИТИЧНО)

- DB-валидаторы (проверки, требующие выборки из БД: уникальность,
  существование связанных сущностей) должны принимать абстракцию
  `IApplicationDbContext` (порт из Domain), а не конкретную коллекцию
  или EF-контекст. Это Dependency Inversion: ядро не знает о EF Core.
- В unit-тестах используется **Moq** для мока `IApplicationDbContext`
  и его свойств (например, `Mock<DbSet<OsaMeeting>>` через
  `.Setup(db => db.OsaMeetings).Returns(mockSet.Object)`).
- Запрещено передавать в DB-валидатор конкретные коллекции
  (`List<T>`, `IEnumerable<T>`) — выборка из БД должна происходить
  внутри валидатора через `IApplicationDbContext`, чтобы тест проверял
  реальную логику, а не «предварительно отфильтрованные» данные.
- Шаблон теста:
  1. Arrange: создаётся `Mock<IApplicationDbContext>`, настраивается
     `Mock<DbSet<T>>` с тестовым списком и провайдером `IQueryable`.
  2. Act: вызывается метод валидатора с mock-объектом.
  3. Assert: проверяется результат (ошибки/успех) через FluentAssertions.

## Безопасность

### Проверка компонентов на уязвимости (CVE)

- При добавлении нового пакета/компонента проверять его на известные
  уязвимости (CVE) через GitHub Security Advisories, Snyk, NVD.
- Результат проверки заносить в `docs/security/vulnerability-check.md`.
- Рекомендовать компоненты с наименьшим количеством уязвимостей в истории;
  при равных показателях — с более активным циклом исправлений.
- При добавлении компонента сообщать пользователю о наличии известных
  уязвимостей (если есть — с оценкой критичности и рекомендацией).

### Чувственные данные

- Не коммитьте секреты в репозиторий
- Используйте `.env` для локальных настроек
- Шифруйте PII данные на уровне приложения

### Аутентификация

- OAuth2 + 2FA для ПК
- JWT Bearer tokens
- ПЭП (СМС) для голосования
- УКЭП (КриптоПро) для председателей

## Git Flow

| Ветка | Назначение | Защита |
|-------|------------|--------|
| `main` | Продакшн | 2 reviewers + CI |
| `develop` | Интеграция | 1 reviewer + CI |
| `feature/*` | Фичи | PR → develop |
| `bugfix/*` | Исправления | PR → develop |
| `hotfix/*` | Экстренные | PR → main + develop |

### Правило: первый коммит

Первый коммит в репозитории всегда имеет название `Initial commit`, независимо от
содержимого. Никаких `feat: ...`, `chore: initial scaffold` и т.п. — только
`Initial commit`.

### Правило: push сразу после commit

Каждый `git commit`, выполненный ассистентом, сопровождается немедленным
`git push` в текущую ветку в том же ответе — без промежуточной остановки
и без отдельного запроса пользователя на публикацию изменений. Коммит без
последующего push считается незавершённым действием.

Исключение — если push завершается конфликтом (например, ветка отстаёт от
`origin`) или требует принудительных операций (`--force`, перезапись истории):
в этом случае синхронизация выполняется безопасным способом (`git pull --rebase`
или аналогичный), а при риске потери чужих изменений — вопрос выносится
пользователю явно, а не решается вслепую.

### Commit Message

```
<type>(<scope>): <description>

feat(meeting): add automatic quorum calculation
fix(voting): correct vote counting logic
docs(api): update voting endpoints
```
