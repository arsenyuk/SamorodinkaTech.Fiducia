## US-001: Настройка юридического лица (Admin Console)

**Как** системный администратор, **я хочу** настроить параметры юридического лица, указав ОПФ и корректный интервал проведения ГОСА, **чтобы** задать базовые параметры организации в системе.

### Критерии приёмки

1) Сценарий входа администратора в Админ-консоль

- [x] Доступ к разделу администрирования разрешён только роли `SYS_ADMIN` (см. также US-002)
- [x] Успешный вход предоставляет доступ к разделу «Юридические лица» (например, `/admin/legal-entities`)

2) Сценарий сохранения данных о ЮЛ

- [x] Форма содержит поля: «Наименование ЮЛ» (обязательно), «ОПФ» (справочник ОКОПФ, по умолчанию пусто), «Интервал проведения ГОСА» (Дата начала, Дата окончания)
- [x] По умолчанию интервал ГОСА установлен стандартный (с 01.03 по 30.06) и поля редактирования интервала заблокированы
- [x] Если выбрана ОПФ = ПАО, поля интервала ГОСА разблокируются для редактирования (любой подинтервал в рамках 01.03–30.06)
- [x] Валидации интервала:
  - Начало ≤ Окончание
  - Начало ≥ 01.03
  - Окончание ≤ 30.06
- [x] При нарушении любого правила отображается соответствующее предупреждение
- [x] «Наименование ЮЛ» не может быть пустым
- [x] Если ОПФ не заполнено, по сохранению отображается предупреждение
- [x] Логика интервала соответствует бизнес-процессу «Определение интервала проведения ГОСА» (см. `docs/business-processes/gosa-window.md`)
  - [x] Для АО/НАО интервал фиксирован 01.03–30.06, для ПАО допускается сокращённый интервал в этих границах

### Негативные сценарии

- [x] **US-001-N1**: Попытка входа в Admin Console пользователем без роли `SYS_ADMIN` → доступ запрещён/редирект
- [x] **US-001-N2**: Пустое «Наименование ЮЛ» → ошибка валидации, сохранение заблокировано
- [x] **US-001-N3**: ОПФ не заполнено при сохранении → предупреждение
- [x] **US-001-N4**: Для АО/НАО пользователь пытается отредактировать интервал → поля недоступны (редактирование запрещено)
- [x] **US-001-N5**: Начало > Окончание → ошибка валидации
- [x] **US-001-N6**: Начало < 01.03 → ошибка валидации
- [x] **US-001-N7**: Окончание > 30.06 → ошибка валидации
- [x] **US-001-N8**: Для ПАО указаны даты вне разрешённого окна 01.03—30.06 → ошибка валидации

## US-012: Просмотр справочника ЮЛ (Admin Console)

**Как** системный администратор, **я хочу** заходить в раздел «Юридические лица» и видеть список ЮЛ со связанной ОПФ (ОКОПФ), **чтобы** управлять реестром организаций.

### Критерии приёмки

1) Страница доступна по `/admin/legal-entities` только для роли `SYS_ADMIN`.
2) В таблице видны поля: `Id`, `Наименование`, `ОПФ` (в формате `код — наименование`).
3) Данные берутся из БД (`legal_entities`) с join на `okopf` по `OkopfCode` → `Code`.

## US-013: Управление ЛОСВ (Admin Console)

**Как** системный администратор, **я хочу** просматривать список ЛОСВ и добавлять новую запись через форму, **чтобы** фиксировать лиц, оказывающих существенное влияние на ЮЛ.

### Критерии приёмки

1) Страница доступна по `/admin/influential-people` только для роли `SYS_ADMIN`.
2) Отображается таблица с полями: `Id`, `ЮЛ`, `ФИО`, `Должность`.
3) Над таблицей расположена форма: выбор ЮЛ (dropdown), поля «ФИО», «Должность», кнопка «Добавить».
4) При нажатии «Добавить» запись сохраняется и появляется в списке без перезагрузки страницы.

Примечание: форма добавления находится над таблицей, админ выбирает ЮЛ, вводит ФИО и должность, нажимает «Добавить» — новая запись появляется в списке.


## US-002: Вход в систему

**Как** директор, **я хочу** войти в систему, выбрав своё ФИО, **чтобы** получить доступ к рабочему пространству.

### Критерии приёмки

- [x] На странице /login отображается dropdown со списком пользователей (Basic)
- [x] Сайдер-меню не содержится на странце входа
- [x] Header содержит только логотип «Fiducia» и название портала
- [x] После выбора ФИО и нажатия «Войти» происходит редирект на главную (приветсвенную) страницу
- [x] На главной (приветсвенной) странице отображается сайдер-меню с разделами
- [x] В header появляется выбранное ФИО и кнопка «Выход»
- [x] Admin Console доступен только для роли SYS_ADMIN
- [x] Устанавливается JWT cookie для серверной сессии (УПД.15)
- [x] Событие входа записывается в аудит-лог (LOGIN_SUCCESS / LOGIN_FAILED)

### Негативные сценарии

- [x] **US-002-N1**: Попытка входа в Admin Console с ролью не-SYS_ADMIN → ошибка «Доступ только для системных администраторов»
- [x] **US-002-N2**: Невалидный логин/пароль → отображение ошибки без указания причины

## US-003: Выход из системы

**Как** директор, **я хочу** выйти из системы через отдельную страницу, **чтобы** безопасно завершить сессию.

### Критерии приёмки

- [x] При нажатии «Отмена» — редирект обратно на главную

## US-011: Принудительное закрытие сессии (УПД.15)

**Как** администратор ИБ, **я хочу** автоматически завершать сессии при бездействии, **чтобы** предотвратить несанкционированный доступ.

### Критерии приёмки

- [x] JS idle-таймер отслеживает активность (мышь, клавиатура, скролл)

- [x] При бездействии > N минут → POST /api/session/logout → очистка → /login
- [x] Endpoint /api/session/logout всегда возвращает 200 OK
- [x] Таймаут передаётся клиенту через GET /api/session/config

### Негативные сценарии

- [ ] **US-011-N1**: Пользователь работает в фоновой вкладке (без активности) → сессия истекает → потеря несохранённых данных

## US-012: Регистрация событий безопасности (РСБ.2)

**Как** администратор ИБ, **я хочу** чтобы все входы/выходы и попытки доступа записывались в аудит-лог, **чтобы** обеспечить соответствие 152-ФЗ.

### Критерии приёмки

- [ ] Запись LOGIN_SUCCESS при успешном входе
- [ ] Запись LOGIN_FAILED при неуспешном входе
- [ ] Запись LOGOUT при выходе
- [ ] Централизованный ISecurityAuditService
- [ ] Записи в PostgreSQL + файловый лог

### Негативные сценарии

- [ ] **US-012-N1**: БД недоступна → аудит пишется только в файл
- [ ] **US-012-N2**: Файловая система недоступна → аудит пишется только в БД
- [ ] **US-012-N3**: Обе системы недоступны → ошибка логируется, но запрос не блокируется
