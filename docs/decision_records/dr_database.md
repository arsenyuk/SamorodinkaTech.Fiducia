## Решения (BDR)

Ниже собраны архитектурные решения по структуре и управлению схемой базы данных. Решения перенесены из `docs/decision_records/architecture.md` и переиндексированы в формат `BDR-{XXX}`. Отсортированы от более абстрактных к более прикладным.

### BDR-001: PostgreSQL как основная БД

Статус: Принято

Контекст: Требование ACID-транзакций, JSON-поддержка, зрелость экосистемы.

Решение: PostgreSQL 16 с Entity Framework Core.

Последствия:
- Позитив: Надёжность, производительность, расширяемость
- Негатив: Ограничения горизонтального масштабирования (решается через шардирование)

### BDR-002: Database‑First как принцип управления схемой БД

Статус: Принято

Контекст: Требуется единый источник истины для структуры БД и исключение «плавающих» миграций при старте приложения. Подход Code‑First с авто‑миграциями усложняет сопровождение, ревью и переносимость между окружениями.

Решение (Database‑First):
- Структура БД ведётся и утверждается в одном каноническом SQL‑файле: `tools/db/01_schema.sql` (DDL). Этот файл описывает полную схему таблиц, индексов, ограничений и расширений.
- Приложение не выполняет миграции схемы при старте (никаких автоматических `EnsureCreated`/`Migrate`). Изменение схемы — только через PR с обновлением `01_schema.sql` (и, при необходимости, вспомогательных скриптов `02_seed.sql`, `03_demo.sql`).
- EF Core используется как Data Access слой, но не как первичный источник эволюции схемы. Модели и конфигурации синхронизируются под фактическую схему из `01_schema.sql`.

Правила именования (схема БД)

1. Таблицы: snake_case, множественное число (например, `legal_entities`).
2. PK: UUID `id`. FK: `<entity>_id` (UUID) или `<classifier>_code` только для суррогатных строковых кодов.
3. Справочники: префикс `ref_` (например, `ref_okopf`), UUID PK + уникальный `code`.
4. Индексы: `ix_<table>_<col1>[_<col2>]`; уникальные: `ux_<table>_<col>`.
5. EF конфигурации обязаны задавать `ToTable` и `HasColumnName` для каждого поля, совпадая с `01_schema.sql`.

Последствия:
- Позитив: единый проверяемый DDL‑артефакт, чёткие ревью изменений схемы, предсказуемый деплой (без скрытых миграций), воспроизводимость окружений.
- Негатив: разработчикам нужно дисциплинированно обновлять SQL‑схему при каждом изменении модели; возможна расхожденность, если править только EF‑модель без DDL.

Практики применения:
1. Любое изменение структуры БД оформляется обновлением `tools/db/01_schema.sql` и проходит code‑review.
2. Инициализация окружения: `01_schema.sql` → `02_seed.sql` → `03_demo.sql`.
3. В коде запрещено вызывать авто‑миграции/автосоздание БД на старте.

### BDR-003: Скриптовая стратегия управления схемой и начальными данными

Статус: Принято

Контекст: Для воспроизводимой инфраструктуры и однозначной инициализации окружений требуются скрипты, создающие схему БД, загружающие справочники и подготавливающие демонстрационные данные. Использование EF‑миграций остаётся допустимым для инкрементальных изменений, но базовая установка должна выполняться единым скриптом, сразу учитывающим UUID‑ключи.

Решение: Вводится три набора скриптов:
1. Скрипт структуры БД (schema): создание расширений (`pgcrypto`), схем, таблиц и индексов. Все PK — `uuid` с `DEFAULT gen_random_uuid()`.
2. Скрипт первоначального наполнения (seed): справочники (например, ОКОПФ), системные записи/роли.
3. Скрипт демонстрационных данных (demo): тестовые пользователи/ФИО с ролями, пример ЮЛ и др. непроизводственные данные.

Примечание: политика генерации UUID уточнена в BDR-005.

Скрипты должны быть идемпотентными (использовать `CREATE ... IF NOT EXISTS`, `ON CONFLICT DO NOTHING`) и версиями фиксироваться в git. Путь размещения будет определён в репозитории (например, `tools/db/01_schema.sql`, `02_seed.sql`, `03_demo.sql`).

Последствия:
- Позитив: детерминируемая инициализация окружений (dev/stage/prod), прозрачная проверка изменений схемы в PR.
- Негатив: дублирование с миграциями EF при параллельном использовании; необходимость дисциплины обновления скриптов при изменениях моделей.

Рекомендации по использованию:
- При «чистой» установке окружения исполнять `01_schema.sql` → `02_seed.sql` → `03_demo.sql`.
- EF‑миграции использовать для инкрементальных апдейтов в ходе разработки; скрипты периодически синхронизировать до актуального состояния схемы.

### BDR-004: UUID (GUID) как обязательный идентификатор записей БД

Статус: Принято

Контекст: В текущей схеме БД часть сущностей использует целочисленные автоинкрементные PK. Требуется унификация идентификаторов для лучшей переносимости данных между окружениями, упрощения интеграций, безопасности раскрытия ключей и возможности офлайн‑генерации без обращения к БД.

Решение: Каждая новая таблица создаётся с первичным ключом типа UUID (GUID). Единая стратегия генерации идентификаторов на проект — на стороне приложений/скриптов (`Guid.NewGuid()` / заранее сгенерированные UUID в SQL‑скриптах). Использование БД‑дефолтов (`gen_random_uuid()`) не рекомендуется (см. BDR‑005) во избежание «скрытой» генерации и для детерминированности скриптов.

Для уже существующих таблиц, созданных ранее с `int` PK, миграция существующих PK не выполняется в рамках этого решения. Новые таблицы и фичи — только UUID PK. Стратегия перевода унаследованных таблиц на UUID (если потребуется) будет описана отдельно.

Последствия:
- Позитив: устойчивые к слияниям идентификаторы, удобная репликация/импорт, отсутствие раскрытия плотности записей; упрощённые ссылки во внешних API.
- Негатив: размер ключей и индексов больше, чем у `int`; необходимость включить `pgcrypto` и поддерживать единые правила генерации.

Рекомендации по реализации:
- EF Core: использовать `Guid` как тип ключа. Значения формировать в приложении (или задавать в SQL‑скриптах явно), а не через БД‑дефолты.
- SQL: тип ключа — `uuid`; дефолт на уровне БД не задавать, или не использовать при вставках.
- Именование столбца: `id uuid PRIMARY KEY`.
- Ссылочные ключи: `uuid` во всех FK, каскады согласно доменной модели.

Не входит в скоуп: Массовая миграция уже созданных `int` PK на `uuid` — будет оформлена отдельным решением при необходимости.

### BDR-005: Явное назначение UUID без gen_random_uuid()

Статус: Принято

Контекст: Для диагностики и воспроизводимости важно, чтобы идентификаторы записей были видимы/фиксированы в артефактах (скриптах/дампах), а не генерировались «скрыто» движком БД при вставке. Это упрощает построение скриптов, ссылающихся на ранее созданные записи, без дополнительных SELECT/LOOKUP по «естественным» ключам.

Решение: В проекте не используем `gen_random_uuid()` (дефолт БД) для генерации PK. UUID формируются явно:
- на уровне приложения (`Guid.NewGuid()`),
- или заранее в SQL‑скриптах (фиксированные значения),
и передаются в INSERT. В SQL‑схеме допускается отсутствие DEFAULT для `id`.

Последствия:
- Позитив: детерминированные, отслеживаемые идентификаторы в скриптах; упрощение генерации связанных INSERT без предварительных запросов к БД.
- Негатив: ответственность за генерацию/уникальность переносится на приложений/скриптов; увеличение объёма скриптов при необходимости фиксировать UUID явно.

Практики:
- В `tools/db/02_seed.sql` и прочих сид‑скриптах при необходимости задавать `id` явно.
- В приложениях — генерировать `Guid.NewGuid()` до вставки.

### BDR-006: Префикс `ref_` для всех справочных таблиц

Статус: Принято

Контекст: Для унификации наименования справочников и однозначного различения доменных сущностей и референсных данных требуется единый стиль именования таблиц справочников.

Решение: Все справочники именуются с префиксом `ref_` (например, `ref_okopf`, `ref_countries`, `ref_roles` при условии, что роли трактуются как справочник). Столбцы идентификаторов в справочниках — `id uuid PRIMARY KEY DEFAULT gen_random_uuid()` (если это не код‑ключ по стандарту), при этом «естественные» кодовые поля (например, `code` в ОКОПФ) сохраняются как уникальные ключи.

Последствия:
- Позитив: упрощённое обнаружение справочников, предсказуемые имена, единые политики бэкапа/кеша.
- Негатив: требуется адаптация текущей схемы (переименование таблиц и ссылок).

Переименование `okopf` → `ref_okopf`:
- Таблица ОКОПФ переименовывается в `ref_okopf`; столбец `code` — уникальный бизнес‑ключ, `id` — суррогатный UUID PK.
- В `tools/db/01_schema.sql` схема отражается с новым именем; загрузочные скрипты используют `ref_okopf`.

### BDR-007: Денормализация — ЮЛ не является внешним ключом для основных доменных таблиц

Статус: Принято

Контекст: Текущая версия платформы («Цифровой Совет Директоров») предназначена для работы в контексте **одного** юридического лица (одной компании). Несмотря на наличие таблиц `legal_entities`, `influential_people` и `legal_entity_board_settings`, образующих изолированный кластер «ЮЛ», основные доменные таблицы (meetings, committees, users, bulletins, agenda_questions и др.) **не ссылаются** на `legal_entities` через FK.

Решение: Принята сознательная денормализация:
- Кластер таблиц, связанных с `legal_entities` (`influential_people`, `legal_entity_board_settings`, `ref_okopf`), образует самодостаточный островок данных о юридическом лице.
- Основные бизнес-таблицы (`meetings`, `committees`, `users`, `bulletins`, `agenda_questions`, `committee_tasks` и т.д.) не содержат колонок `legal_entity_id` и не ссылаются на `legal_entities`.
- Данные, которые в нормализованной модели потребовали бы FK на `legal_entities`, либо не хранятся (подразумевается единственное ЮЛ в системе), либо зафиксированы на уровне бизнес-логики и UI.

Последствия:
- Позитив: упрощённые запросы (нет лишних JOIN), меньше ограничений целостности на уровне БД, быстрее разработка в текущем одно-компанийном режиме.
- Негатив: при переходе к мульти-компанийному режиму потребуется нормализация — добавление FK `legal_entity_id` в основные таблицы, миграция данных, обновление бизнес-логики.

Условия пересмотра: Решение должно быть пересмотрено при появлении требования поддержки нескольких независимых юридических лиц в рамках одной инсталляции платформы.

### BDR-008: Денормализация role_code в board_member_appointments

Статус: Принято

Контекст: Таблица `board_member_appointments` хранит назначения членов СД на должности. Ссылка на справочник ролей идёт через `role_id` (FK → `ref_board_roles`). Однако частые операции — фильтрация членов по роли (например, «найти всех секретарей»), проверка уникальности, отчёты — требуют дополнительного JOIN к `ref_board_roles` только ради значения `code`.

Решение: Добавлена денормализованная колонка `role_code VARCHAR(20) NOT NULL`, дублирующая `ref_board_roles.code` для строки роли. При вставке/обновлении назначения код роли записывается явно (извлекается из `ref_board_roles` по `role_id`). Колонка не является внешним ключом к `ref_board_roles(code)` — это осознанная денормализация, а не ссылка на отдельный справочник.

Последствия:
- Позитив: запросы с фильтрацией по роли не требуют JOIN к `ref_board_roles`; упрощённые проверки целостности на уровне приложения (например, «временный председательствующий не более одного» проверяется без подзапроса к справочнику).
- Негатив: денормализованные данные могут расходиться со справочником при изменении кода роли в `ref_board_roles`; требуется обеспечить синхронное обновление в коде создания/изменения назначений.

Практики:
- При создании `BoardMemberAppointment` всегда заполнять `RoleCode` значением из `boardRoles.FirstOrDefault(r => r.Id == row.RoleId)?.Code`.
- При изменении кода роли в справочнике — проверить и при необходимости обновить `role_code` в существующих назначениях.
