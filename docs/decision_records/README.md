# Decision Records (ADR)

Цель: фиксировать ключевые архитектурные и продуктовые решения проекта с мотивацией, альтернативами и последствиями. DR помогает новым участникам быстро понять «почему так», а не только «как сделано».

Что размещается здесь
- Архитектурные решения (ADR): технологии, границы модулей, протоколы взаимодействия, критичные политики (логирование, аудит, хранение файлов и т.п.).
- Проектные решения уровня процесса/доков, влияющие на реализацию.

Правила ведения
1. Один DR — одно решение. Не смешивать несколько тем в одном файле.
2. Имена файлов в стиле `dr_<краткая_тема>.md` латиницей. Примеры: `dr_architecture.md`, `dr_database.md`.
3. Статус решения: `proposed` → `accepted` → `deprecated`/`superseded`.
4. Изменение/отмена решения создаёт новый DR, который ссылается на предыдущий (`supersedes` / `superseded by`).
5. Все публичные утверждения должны быть в терминах репозитория (таблицы, классы, пути, конфиги) и законов РФ, если применимо.

Рекомендуемая структура DR
- Контекст: проблема/обстоятельства, ограничения, допущения.
- Решение: краткое формулирование и границы применимости.
- Альтернативы: что рассматривали и почему отклонено.
- Последствия: плюсы, минусы, риски, влияние на код/инфру/команду.
- Реализация: где в коде/конфиге зафиксировано (пути, типы, ключи).
- Связанные документы/нормативка: ссылки на `AGENTS.md`, `docs/…`, ФЗ/ГОСТ и т.п.
- Статус: proposed/accepted/deprecated/superseded (+даты).

Шаблон (скопируйте при создании нового DR)
```
# DR: <краткое название>

Статус: proposed | accepted | deprecated | superseded (ссылка)
Дата: YYYY-MM-DD
Автор: <ФИО/ник>

## Контекст
<почему это решение понадобилось>

## Решение
<что именно принято, границы, инварианты>

## Альтернативы
- <вариант 1> — почему отклонён
- <вариант 2> — почему отклонён

## Последствия
+ Плюсы: …
- Минусы/риски: …
Влияние: код/конфиг/инфра/процессы

## Реализация
- Код/конфиг: <пути к файлам/классам/разделам конфигурации>
- Тесты/валидации: <как проверяем>

## Связанные документы
- <ссылки на AGENTS.md, другие DR, бизнес-процессы, ФЗ/ГОСТ>
```

Примечания
- Локальные настройки и параметры окружения не оформляются DR, если не меняют архитектурных инвариантов.
- Для настроек логирования, путей хранилищ, политик аудита — решения фиксируются DR и дублируются в `AGENTS.md` (раздел «Архитектурные принципы») как обязательные правила.
