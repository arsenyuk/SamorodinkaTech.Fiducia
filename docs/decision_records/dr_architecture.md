# Архитектурные решения (ADR)

## ADR-001: Выбор монолитной архитектуры

**Статус**: Принято

**Контекст**: Требование изолированного Enterprise-решения без внешних интеграций на первом этапе.

**Решение**: Модульный монолит с выделенными сервисами: Auth, Meeting, Voting, Committee, Document, Audit.

**Последствия**:
- Позитив: Простота развёртывания, транзакционность, контроль версий
- Негатив: Ограничения масштабирования (решается в Фазе 3)

## ADR-002: CQRS vs CRUD — терминологическое разграничение

**Статус**: Принято

**Контекст**: В коде и документации термин "CQRS" ошибочно применялся к обычным CRUD-репозиториям/сервисам (методы Create/Read/Update/Delete поверх одной модели и одного DbContext), что размывало паттерн и вводило команду в заблуждение относительно фактической архитектуры модуля.

**Решение**: Зафиксировано строгое разграничение:
- **CRUD** — класс/сервис с методами `CreateAsync`, `GetAsync`, `UpdateAsync`, `DeleteAsync` над одной моделью и одним хранилищем. Используется в большинстве модулей системы по умолчанию.
- **CQRS** — применяется только там, где есть логическое разделение верхнеуровневых модулей Command и Query (разные namespace/классы/handlers), с отдельными моделями для чтения и записи, а в перспективе — раздельными инстансами БД (Read replica) с синхронизацией через Domain Events.
- Называть CRUD-реализацию "CQRS" в коде, PR, документации и коммитах запрещено.

**Последствия**:
- Позитив: Точность технической коммуникации, корректные ожидания от структуры модуля при code review
- Негатив: Требуется ревизия существующих модулей и документации, где термин использовался некорректно

## ADR-003: Явное отображение бизнес/юридического уровня в архитектуре

**Статус**: Принято

**Контекст**: Техническая документация архитектуры (диаграммы компонентов, модулей, потоков данных) не давала явного ответа на вопрос, где и как отображается соответствие системы нормам федеральных законов (208-ФЗ, 14-ФЗ, 63-ФЗ, 152-ФЗ). Требования законодательства были неявно «растворены» в описании модулей, что затрудняло аудит соответствия и проверку полноты покрытия правовых норм технической реализацией.

**Решение**: Вводится отдельный, явно выделенный **бизнес/юридический уровень трассируемости**, не подменяющий и не смешивающийся с техническими слоями (Api/Application/Domain/Infrastructure). Он представлен двумя артефактами:
1. Сводная таблица соответствия «норма закона → модуль/сервис → статус реализации» в разделе «Бизнес-уровень» текущего документа (`docs/architecture.md`).
2. Детальная матрица трассируемости в отдельном документе [`docs/legal-traceability.md`](legal-traceability.md), включающая конкретные статьи законов, ответственные компоненты и статус покрытия.

Ведение и актуализация матрицы при изменении законодательства — обязанность владельца соответствующего модуля; ссылки на ADR-004 фиксируются в обоих документах для однозначной трассируемости.

**Последствия**:
- Позитив: Явная проверяемость соответствия 208-ФЗ/14-ФЗ/63-ФЗ/152-ФЗ техническим компонентам; упрощён юридический аудит и онбординг новых разработчиков/юристов
- Негатив: Дополнительный документ (`legal-traceability.md`) требует синхронизации при рефакторинге модулей и изменении законодательства

## ADR-004: Redis для кэширования

**Статус**: Принято

**Контекст**: Высокая нагрузка на чтение, необходимость быстрого доступа к часто запрашиваемым данным.

**Решение**: Redis Cluster для кэширования результатов проверок и сессий.

**Последствия**:
- Позитив: Снижение нагрузки на БД, ускорение отклика
- Негатив: Сложность инвалидации кэша, стоимость памяти

## ADR-005: Хранение настроек логирования исключительно в конфигурационном файле

**Статус**: Принято

**Контекст**: В рамках задачи о настраиваемых путях логирования реализация вышла за рамки исходного запроса пользователя (backend-конфигурация) и затронула административный UI, где стало возможным управлять путями/параметрами логирования. Это создало риск смешения инфраструктурной конфигурации (уровни логирования, пути к файлам, форматы, ротация) с прикладными данными в БД/UI: потеря единого источника истины, отсутствие версионирования настроек через git, несогласованность между окружениями (dev/stage/prod), избыточная административная поверхность для параметров, которые по своей природе относятся к деплою, а не к бизнес-данным.

**Решение**: Все настройки логирования (уровни логирования, пути к файлам логов, форматы, sink'и, ротация) хранятся исключительно в конфигурационном файле (`appsettings.json` / `appsettings.{Environment}.json`) или переменных окружения и считываются через стандартный механизм конфигурации ASP.NET Core (`IOptions<TLoggingOptions>` / `IConfiguration`). Хранение этих настроек в базе данных, а также их редактирование через пользовательский интерфейс (Admin Console, Board Portal) запрещено, если это не запрошено пользователем явно и отдельно от прочих задач. Соответствующее процессное правило зафиксировано в `AGENTS.md` (раздел «Архитектурные принципы» → «Конфигурация логирования (КРИТИЧНО)»); данный ADR — архитектурное обоснование этого правила.

**Последствия**:
- Позитив: единый источник истины для конфигурации логирования; совместимость со стандартными DevOps-практиками (переменные окружения в Docker/CI); версионирование через git; отсутствие лишней административной поверхности для инфраструктурных параметров.
- Негатив: изменение параметров логирования требует передеплоя/перезапуска приложения (либо `reloadOnChange` конфигурации), а не доступно оперативно через UI администратору без выхода в конфигурационные файлы или переменные окружения хоста.

## ADR-006: Отдельный layout для неавторизованных страниц

**Статус**: Принято

**Контекст**: Обнаружен дефект — страница `/login` в Admin Console отображала сайдбар с навигацией (`NavMenu`) внутренней функциональности, поскольку Blazor Router (`App.razor`) применяет единый `DefaultLayout="MainLayout"` ко всем страницам, включая страницы входа/выхода, не требующие авторизации.

**Решение**: В каждом Blazor-портале (AdminConsole, BoardPortal) вводится отдельный layout (например, `Shared/EmptyLayout.razor`) без `NavMenu` и ссылок на защищённые разделы. Страницы `/login`, `/logout` и другие неавторизованные страницы (восстановление доступа, соглашение о ПЭП) подключают его явно через директиву `@layout`. `DefaultLayout` в `App.razor` остаётся `MainLayout` — используется только авторизованными страницами.

**Последствия**:
- Позитив: Устранена утечка навигации по защищённым разделам для неавторизованных пользователей; повышена соответствие требованиям ИБ (152-ФЗ)
- Негатив: Необходимо поддерживать два layout-шаблона на портал и следить за корректным подключением `@layout` на новых неавторизованных страницах

## ADR-007: Login = Index, Main = Authenticated Home

**Статус**: Принято

**Контекст**: При старте приложений (Admin Console, Board Portal) отображалась главная страница до аутентификации. Это нарушает требования ИБ: до входа должен показываться только экран ввода учётных данных.

**Решение**:
- Страница `Index (/)` — это экран входа (Login) без сайдбара (AuthLayout).
- После успешной аутентификации выполняется явный переход на `/main`.
- Страница `/main` — аутентифицированная главная (дашборд). В Admin Console — только для роли `SYS_ADMIN`.
- Router использует AuthorizeRouteView/RedirectToLogin, чтобы незалогиненные всегда оказывались на `/login`.

**Затронутые компоненты**:
- Admin Console:
  - Pages/Index.razor → Login
  - Pages/Main.razor → Дашборд (перенесён)
  - Shared/AuthLayout.razor → страницы входа/выхода без меню
  - Shared/NavMenu.razor → ссылка «Дашборд» указывает на `/main`
- Board Portal:
  - Pages/Login.razor → редирект на `/main` после входа
  - Pages/Main.razor → главная после входа
  - Index остаётся Login, аналогично админке

**Последствия**:
- Исключён показ чувствительной служебной информации до входа.
- Поведение ссылок и прямых переходов стало предсказуемым: `/` всегда логин, `/main` — главная после входа.

## ADR-009: Серверная валидация — изолированный валидатор для каждой формы сохранения (BDR‑008)

**Статус**: Принято  
**Дата**: 2026-07-15

### Контекст
В Admin Console каждая форма сохранения данных требует проверки корректности вводимых значений на стороне сервера. Клиентская валидация дублирует проверки и не гарантирует согласованности.

### Решение
1. Для каждой формы сохранения в Admin Console создаётся изолированный серверный валидатор — статический класс/метод в `Domain.Validation`, не зависящий от БД, DI-контейнера или HTTP-контекста.
2. Валидатор принимает plain‑data модель (record) и возвращает результат со списком ошибок.
3. `Save()` вызывает валидатор первым действием, до любых операций с БД.
4. Реализация: `LegalEntityValidator` в `src/Domain/Validation/`.

### Альтернативы
- Data Annotations — привязаны к EF-сущностям, не работают без DbContext.
- FluentValidation — добавляет внешнюю зависимость без необходимости.
- Только клиентская проверка — не защищает от прямых вызовов API.

### Последствия
- Позитив: 48 тестов, полная изоляция от окружения, тестируется без моков.
- Негатив: дублирование с клиентским `CanSave` (для UX); нужна синхронизация.

### Реализация (пример)
- Модель: `src/Domain/Validation/LegalEntitySaveValidationModel.cs` (record)
- Результат: `src/Domain/Validation/LegalEntitySaveValidationResult.cs`
- Валидатор: `src/Domain/Validation/LegalEntityValidator.cs` — `Validate(Model) → Result`
- Тесты: `tests/.../Validation/LegalEntityValidatorTests.cs` — 48 тестов
- Интеграция: `LegalEntities.razor` → `Save()`

### Правило для новых форм
При создании формы сохранения в Admin Console:
1. Создать `Domain/Validation/` с моделью, результатом и валидатором
2. Написать unit‑тесты на все позитивные и негативные сценарии
3. В `Save()` вызвать `Validator.Validate(model)` первым шагом
4. При ошибках — прервать сохранение, показать `saveError`

### Логирование валидации
- Вызов валидатора логируется как **Debug**: `LogDebug("Серверная валидация {EntityType} {Id} ...")`  
  Позволяет отследить в логах факт выполнения проверки без захламления INFO-уровня.
- Обнаруженные ошибки логируются как **Warning**: `LogWarning("Валидация не пройдена для {EntityType} {Id}: {Errors}")`  
  Фиксирует попытки сохранить некорректные данные (возможный баг UI или злонамеренный вызов).
- Логирование встроено в каждый метод `Save()` после вызова валидатора.

## ADR-010: ОСА — умолчания полей из последнего ОСА и очистка при отсутствии СД

**Статус**: Принято  
**Дата**: 2026-07-15

### Контекст
Форма редактирования ОСА содержит поля «Количество акционеров» и «Количество участников СД», которые переносятся из режима ЮЛ. При создании нового ОСА эти поля не заполнены. При отсутствии Совета директоров у организации поля СД не применимы и должны очищаться.

### Решение
1. **Умолчания**: если поле не заполнено в текущем ОСА — берётся значение из последнего (по `created_at`) ОСА с заполненным полем. Если предыдущих ОСА нет: для акционеров — 1, для участников СД — минимальное по ОКОПФ.
2. **Очистка при отсутствии СД**: если `LegalEntityBoardSettings` не существует (СД не создан), блок «Параметры СД» скрывается с формы, а при сохранении поля `shareholders_count` и `board_member_number` обнуляются (`null`).

### Последствия
- Позитив: пользователю не нужно повторно вводить данные, которые не изменились с прошлого ОСА; согласованность данных при отключённом СД
- Негатив: дополнительные запросы к БД при открытии формы редактирования (до 2 запросов к `osa_meetings`)

## ADR-011: Интеграция с LDAP/AD-каталогом для синхронизации состава Совета директоров

**Статус**: Принят  
**Дата**: 2026-07-17

### Контекст

При подготовке ОСА администратор формирует состав Совета директоров. Данные о директорах уже существуют в корпоративном LDAP/AD-каталоге (ФИО, должность, email, телефон, членство в группе «Совет директоров»). Повторный ручной ввод этих данных в Admin Console дублирует информацию, создаёт риск расхождений и замедляет процесс.

Требуется: (1) читать кандидатов из каталога в режиме редактирования СД, (2) привязывать LDAP-учётные записи к типам директоров справочника `ref_board_member_types`, (3) контролировать уникальность — одна учётная запись не может быть назначена дважды в одном составе СД.

Ограничение: LDAP-сервер может быть недоступен на этапе разработки и локального тестирования (Apple Silicon, отсутствие инфраструктуры). Интеграция должна быть опциональной и не блокировать запуск приложения.

### Решение

1. **Опциональность**: секция `Ldap.Enabled` в `appsettings.json` управляет подключением. По умолчанию `false` — сервисы не регистрируются в DI, приложение стартует без LDAP.

2. **Слоистая архитектура (SOLID)**:
   - `ILDapService` (Domain) — низкоуровневые LDAP-операции: аутентификация, поиск, членство в группах.
   - `LdapService` (Infrastructure) — реализация через `System.DirectoryServices.Protocols` (LDAP v3, совместимость с AD и OpenLDAP).
   - `IBoardMemberLdapService` (Domain) — бизнес-уровень: маппинг LDAP-должностей (`title`) на коды `ref_board_member_types`, контроль уникальности (`IsDuplicate`, `FindDuplicates`).
   - `BoardMemberLdapService` (Infrastructure) — композиция над `ILDapService`.

3. **Docker-демо**: `docker-compose.ldap.yml` с OpenLDAP (`osixia/openldap`), предзаполненный seed.ldif с организацией «ПАО Брянский арсенал» и 6 членами СД в группе `cn=BoardOfDirectors`.

4. **Независимость от LDAP в тестах**: mock-объекты (`MockLdapService`, `MockBoardMemberLdapService`) с in-memory хранилищем — 34 unit-теста покрывают все операции без поднятого сервера.

### Альтернативы

- **Прямое использование `System.DirectoryServices` в Blazor-компонентах** — отклонено: нарушает чистую архитектуру, невозможно тестировать без LDAP, смешивает UI и инфраструктуру.
- **Только AD-провайдер (существующий `ActiveDirectoryProvider`)** — отклонено: он выполняет только аутентификацию, не поддерживает поиск, группы и маппинг на бизнес-сущности.
- **FreeIPA / Samba AD DC** — отклонено для MVP: избыточная сложность развёртывания, OpenLDAP протокольно совместим и достаточен для разработки.

### Последствия

- **Позитив**: единый источник данных о директорах (LDAP); автоматический маппинг должностей на типы СД; контроль дублирования учётных записей; опциональность — не блокирует разработку.
- **Негатив**: зависимость от доступности LDAP в production (смягчено опциональным флагом); `System.DirectoryServices.Protocols` требует Windows- или Linux-окружения с LDAP-клиентом.
- **Влияние**: код — `src/Domain/Interfaces/ILDapService.cs`, `IBoardMemberLdapService.cs`; конфиг — секция `Ldap` в `appsettings.json` обоих порталов; инфра — `docker-compose.ldap.yml`.

### Реализация

- Интерфейсы: `src/Domain/Interfaces/ILDapService.cs`, `IBoardMemberLdapService.cs`
- Модели: `src/Domain/Models/Ldap/LdapUser.cs`, `BoardMemberCandidate.cs`
- Реализации: `src/Infrastructure/Services/LdapService.cs`, `BoardMemberLdapService.cs`
- DI: `Program.cs` обоих порталов — `if (Ldap:Enabled)` → Singleton
- Конфиг: `appsettings.json` → `Ldap.Enabled`, `Server`, `Port`, `BaseDn`, `BindUser`, `BindPassword`, `BoardGroupDn`
- Docker: `docker-compose.ldap.yml` + `tools/ldap/seed.ldif`
- Тесты: `tests/.../Mocks/MockLdapService.cs`, `MockBoardMemberLdapService.cs` + `LdapServiceTests.cs` (11 тестов), `BoardMemberLdapServiceTests.cs` (12 тестов)

## ADR-022: Все настройки внешних интеграций — в конфигурационном файле, не в коде

**Статус**: Принято

**Контекст**: В проекте последовательно добавляются внешние интеграции (TrueConf, МТС Линк, LDAP, файловое хранилище, СПАРК). Для каждой из них de facto сложилась практика хранения настроек в `appsettings.json`, но формального архитектурного решения, закрепляющего этот принцип как общий для всех интеграций, не было. ADR-005 зафиксировал аналогичное правило для логирования — данный ADR распространяет его на все настройки внешних систем.

**Решение**: Все настройки внешних интеграций (URL API, ключи доступа, таймауты, флаги включения/отключения, параметры подключения) хранятся **исключительно** в конфигурационном файле (`appsettings.json` / `appsettings.{Environment}.json`) или переменных окружения. Запрещено:

- Хардкодить URL, ключи или endpoints в коде (включая значения по умолчанию в конструкторах клиентов)
- Хранить настройки интеграций в базе данных
- Редактировать настройки интеграций через UI (Admin Console / Board Portal)

Допустимые исключения:
- Значения по умолчанию, обеспечивающие работу приложения при отсутствии секции в конфиге (например, `Enabled: false` для опциональных интеграций), могут быть заданы в коде через `??` с явным комментарием
- Endpoint'ы API, специфичные для конкретной версии (`/api/v1/...`), жёстко заданы в реализации клиента — они являются частью контракта, а не настройкой

**Реализация в коде**: настройки считываются через:
- `IConfiguration` с индексатором (`config["Spark:ApiKey"]`) — для простых случаев
- `IOptions<T>` с привязкой секции (`builder.Configuration.GetSection("Spark").Get<SparkOptions>()`) — рекомендуется для интеграций с 3+ параметрами

**Последствия**:
- Позитив: единый источник истины, версионирование через git, совместимость с DevOps (переменные окружения в Docker/CI/CD), отсутствие «магических» значений в коде
- Негатив: изменение настроек требует передеплоя (либо `reloadOnChange`); для оперативных изменений без перезапуска может потребоваться механизм hot-reload конфигурации

**Охват**: правило действует для всех текущих и будущих интеграций:

| Интеграция | Секция в appsettings.json | Статус |
|---|---|---|
| TrueConf | `TrueConf` | Соответствует |
| МТС Линк | `MtsLink` | Соответствует |
| LDAP | `Ldap` | Соответствует |
| Файловое хранилище | `FileStorage` | Соответствует |
| СПАРК | `Spark` | Соответствует |
| Аудит безопасности | `SecurityAudit` | Соответствует |
| Сессии | `Session` | Соответствует |
| Логирование | `Serilog` + `ApplicationLogging` | Соответствует (ADR-005) |

## ADR-023: Lazy Initialization для внешних интеграций — приложение не падает при ошибках конфигурации

**Статус**: Принято  
**Дата**: 2026-07-19

### Контекст

В Program.cs обоих порталов есть точки, где ошибка в конфигурации приводит к аварийному завершению приложения:
- `throw new InvalidOperationException` — при несоответствии `Auth:Method=LDAP` и `Ldap:Enabled=false`
- `int.Parse(cfg["Port"] ?? "389")` — при невалидном значении порта
- `throw new InvalidOperationException("Session:JwtSecret is not configured")` — при отсутствии секрета

Для enterprise-приложения в изолированном контуре падение всего приложения из-за неверной настройки одной опциональной интеграции неприемлемо. Приложение должно стартовать и работать в деградированном режиме, сигнализируя о проблеме в логах.

### Решение

1. **Конструкторы клиентов не выполняют сетевых соединений** — отложенная инициализация (lazy): подключение к внешнему сервису происходит только при первом вызове метода, а не в конструкторе. Этот принцип уже соблюдается для LdapService, SparkApiClient, FileStorage.

2. **Ошибки конфигурации не крашат приложение**:
   - `throw new InvalidOperationException` при несоответствии параметров → `Log.Warning` + fallback на значение по умолчанию или отключение интеграции
   - `int.Parse(...)` без проверки → замена на `int.TryParse` с fallback-значением
   - Исключение: критичные для безопасности параметры (JWT-секрет в production) сохраняют `throw`, но с проверкой окружения — в dev-режиме используется dev-ключ

3. **Опциональные интеграции регистрируются только при явном флаге** — уже соблюдается через `Ldap:Enabled`, `Spark:ApiKey = ""` (пустой ключ = интеграция отключена).

4. **Для будущих интеграций с тяжёлой инициализацией** — рекомендуемый паттерн: `Lazy<T>` в DI, где конструктор `Lazy<T>` обёрнут в try-catch с fallback на no-op реализацию.

### Конкретные исправления в Program.cs (оба портала)

| Строка | Проблема | Исправление |
|--------|----------|-------------|
| LDAP Auth: `throw new InvalidOperationException` | Крах при `Auth:Method=LDAP` + `Ldap:Enabled=false` | `Log.Warning` + переключение на `BasicProvider` |
| LDAP Service: `int.Parse(cfg["Port"] ?? "389")` | `FormatException` при невалидном порте | `int.TryParse` с fallback `389` |
| JWT: `throw new InvalidOperationException(...)` | Крах при отсутствии `Session:JwtSecret` | Dev: dev-ключ + `Log.Warning`; Prod: `throw` (сохранено для безопасности) |

### Альтернативы

- **Оставить `throw` для всех ошибок конфигурации** — отклонено: неоправданно жёстко для enterprise-приложения, где одна опциональная интеграция не должна блокировать весь старт.
- **Полностью убрать все `throw`, включая JWT в production** — отклонено: JWT-секрет критичен для безопасности, его отсутствие в production должно прерывать запуск.

### Последствия

- **Позитив**: приложение стартует даже при ошибках в конфигурации опциональных интеграций; администратор видит предупреждение в логах и может исправить конфиг без остановки системы; dev-окружение не требует обязательного JWT-секрета.
- **Негатив**: ошибка конфигурации может остаться незамеченной до первого использования интеграции (решается мониторингом логов на `Warning`).
- **Риск**: в dev-режиме JWT подписывается предсказуемым ключом — при развёртывании dev-сборки в публичной сети потребуется явно задать `Session:JwtSecret`.
